# Story 14.1 — Dual Weight Schema & Data Layer (weight_mode + manual_weight)

**Epic:** 14 — Dual Weight System
**Status:** Ready for Review
**Effort:** M (3 points)
**Dependencies:** 11.2 (wallet_id migration complete — this migration comes after)
**PRD Feature:** F-028

## Description

As a developer,
I want `weight_mode` and `manual_weight` columns added to the `assets` table with the XOR resolution logic in the data layer,
so that the rebalancing algorithm can use either manual notes or questionnaire scores per asset without any changes to the L3 algorithm itself.

The XOR resolution happens in the data layer (when building `L3AssetInput`), keeping the L3 algorithm pure and unchanged (ADR-015).

[Source: PRD-v2 §3 Module 15, Feature F-028; ADR-015 (Dual Weight Mode)]

## Acceptance Criteria

- [ ] Given the migration is applied, when the `assets` table is inspected, then it has `weight_mode TEXT NOT NULL DEFAULT 'questionnaire' CHECK (weight_mode IN ('manual', 'questionnaire'))`
- [ ] Given the migration is applied, when the `assets` table is inspected, then it has `manual_weight NUMERIC DEFAULT 0 CHECK (manual_weight >= -10 AND manual_weight <= 11)`
- [ ] Given an existing asset, when the migration runs, then its `weight_mode` is set to `'questionnaire'` (backward-compatible default)
- [ ] Given the data layer function `buildL3Input(asset, assetScore)`, when `weight_mode = 'manual'`, then it returns `manual_weight` as the `score` field
- [ ] Given the data layer function `buildL3Input(asset, assetScore)`, when `weight_mode = 'questionnaire'`, then it returns `assetScore.total_score` as the `score` field (or 0 if no score)
- [ ] Given the `distributeL3` function, when inspected, then it has NOT been modified (L3 remains pure per ADR-015)
- [ ] Given a group with mixed weight modes (some manual, some questionnaire), when scores are normalized, then `normalizeScores` correctly handles both (same scale -10 to +11)
- [ ] Given a mode switch from manual to questionnaire, when the asset is saved, then the `manual_weight` value is preserved in the DB (not cleared) for potential reversion
