# Story 10.1 — Fix /api/asset-types HTTP 500 Bug

**Epic:** 10 — Bug Fix & Preparation
**Status:** Ready for Review
**Effort:** S (2 points)
**Dependencies:** 1.1 (schema), 1.2 (RLS policies)
**PRD Feature:** F-021B

## Description

As a user,
I want the `/api/asset-types` endpoint to correctly return my asset types list,
so that the "Register Asset" button works and I can start configuring my portfolio.

The current endpoint executes an INSERT instead of a SELECT, causing a 500 error because RLS correctly blocks the unauthorized INSERT. The fix must change the logic to SELECT only and return the user's asset types.

[Source: PRD-v2 §3 Module 10, Feature F-021B]

## Acceptance Criteria

- [ ] Given an authenticated user with asset types, when GET `/api/asset-types` is called, then HTTP 200 is returned with an array of the user's asset types
- [ ] Given an authenticated user with no asset types, when GET `/api/asset-types` is called, then HTTP 200 is returned with an empty array `[]`
- [ ] Given an unauthenticated request, when GET `/api/asset-types` is called, then HTTP 401 is returned
- [ ] Given the fixed endpoint, when the "Register Asset" page loads, then the "Register Asset" button is enabled when all required fields are filled
- [ ] Given the endpoint response, when inspected, then each asset type object contains `id`, `name`, `target_pct`, `sort_order`

## Dev Notes

- **Primary file:** API route handler for `/api/asset-types` (likely in azimute-blog `src/pages/api/`)
- **Root cause:** Route executes `INSERT` instead of `SELECT` — RLS correctly blocks unauthorized INSERT
- **Fix:** Change to `SELECT * FROM asset_types WHERE user_id = auth.uid()` via Supabase client
- **Constraints:** Must use Supabase client with auth context from request
