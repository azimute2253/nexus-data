# Story 11.3 — Wallet Data Access Layer (CRUD + Isolation)

**Epic:** 11 — Multi-Wallet Foundation
**Status:** Draft
**Effort:** M (3 points)
**Dependencies:** 11.2 (wallet_id FK must exist on all tables)
**PRD Feature:** F-022, F-024

## Description

As a developer,
I want a data access layer for wallets (CRUD operations) with wallet-scoped data functions,
so that all data queries are automatically filtered by the active wallet and the application correctly isolates data between wallets.

This includes updating all existing data functions (asset types, groups, assets, scores, contributions) to accept and filter by `wallet_id`.

[Source: PRD-v2 F-022, F-024; ADR-014 (Data Isolation — RLS by user_id + app-layer filter by wallet_id)]

## Acceptance Criteria

- [ ] Given the wallet client module, when inspected, then it exports: `getWallets(userId)`, `getWallet(id)`, `createWallet(name)`, `updateWallet(id, name)`, `deleteWallet(id)`
- [ ] Given `createWallet(name)`, when called with a valid name, then a new wallet is created and returned with `id`, `user_id`, `name`, `created_at`
- [ ] Given `deleteWallet(id)`, when called, then the wallet and ALL associated data (6 tables) are cascade-deleted
- [ ] Given all existing data functions, when inspected, then they accept `wallet_id` as a required parameter and filter queries by it
- [ ] Given wallet A with 3 asset types and wallet B with 2 asset types, when `getAssetTypes(walletId_A)` is called, then only wallet A's 3 types are returned
- [ ] Given a wallet name update, when `updateWallet(id, newName)` is called, then the name is updated and the response confirms the change
- [ ] Given the wallet name validation, when a name < 1 char or > 50 chars is provided, then the function throws a validation error
