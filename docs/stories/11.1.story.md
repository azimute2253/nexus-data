# Story 11.1 — Create `wallets` Table with RLS

**Epic:** 11 — Multi-Wallet Foundation
**Status:** Ready for Review
**Effort:** S (2 points)
**Dependencies:** 1.1 (schema), 1.2 (RLS policies)
**PRD Feature:** F-022

## Description

As a developer,
I want a `wallets` table created with proper schema, constraints, and RLS policies,
so that the multi-wallet architecture has a solid foundation for all subsequent V2 features.

[Source: PRD-v2 §3 Module 11, Feature F-022; ADR-010 (Multi-Wallet Schema)]

## Acceptance Criteria

- [ ] Given the migration is applied, when the `wallets` table is inspected, then it has columns: `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`, `user_id UUID NOT NULL REFERENCES auth.users(id)`, `name TEXT NOT NULL`, `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`
- [ ] Given the RLS is enabled on `wallets`, when a user queries, then they can only see/modify their own wallets (`auth.uid() = user_id`)
- [ ] Given a wallet name, when validated, then it must be between 1 and 50 characters (non-empty, length-limited)
- [ ] Given the TypeScript types file, when inspected, then it includes `Wallet`, `WalletInsert`, `WalletUpdate` interfaces matching the schema
- [ ] Given the migration file, when re-run, then it completes without errors (idempotent via `IF NOT EXISTS`)

## Test Requirements

| Test | Input | Expected | Validation |
|------|-------|----------|------------|
| T11.1.1 | Apply migration | `wallets` table created | `\dt wallets` succeeds |
| T11.1.2 | INSERT wallet for user A, SELECT as user B | Empty result | RLS blocks cross-user access |
| T11.1.3 | INSERT wallet with empty name | Constraint violation | Error returned |
| T11.1.4 | INSERT wallet with 51-char name | Constraint violation | Error returned |
| T11.1.5 | Re-run migration | No error | Idempotent |
