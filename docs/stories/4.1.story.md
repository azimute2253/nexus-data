# Story 4.1 — L1 Distribution (Type-Level Rebalancing)

**Phase:** 4 — Rebalancing Algorithm
**Status:** Draft
**Effort:** M
**Dependencies:** 1.1 (schema/types for portfolio data structures), ADR-004 (Rebalancing Algorithm Architecture)

## Description

As a user,
I want the rebalancing algorithm to distribute my aporte across the 10 asset types based on their target % vs actual allocation,
so that underweight classes receive proportionally more and overweight classes (like FIIs at 65% vs 15% target) receive R$0.

L1 is the first level of the cascade. It computes the deficit per type (target value - actual value), zeroes out overweight types, and distributes the contribution proportionally among underweight types.

[Source: PRD §3 Module 2, Feature F-002 AC1/AC3/AC8]

## Acceptance Criteria

- [ ] Given an aporte of R$12,000 and 10 asset types with target %, when `distributeL1()` runs, then the total distributed equals R$12,000
- [ ] Given FIIs at 65% actual vs 15% target (overweight), when L1 distributes, then FIIs receive exactly R$0
- [ ] Given multiple underweight types, when L1 distributes, then amounts are proportional to each type's deficit (target_value - actual_value)
- [ ] Given all types are overweight, when L1 distributes, then the entire aporte goes to the least-overweight type (graceful degradation)
- [ ] Given the function signature, when called, then it accepts: `types: AssetType[]`, `prices: PriceMap`, `exchangeRates: RateMap`, `contribution: number` and returns `L1Result[]`
- [ ] Given the function, when inspected, then it is a pure function with no side effects (no DB calls, no API calls, no Date.now())
- [ ] Error: Given contribution = 0, when L1 runs, then all allocations are R$0
