# Story 11.2 — Add `wallet_id` FK to All Data Tables

**Epic:** 11 — Multi-Wallet Foundation
**Status:** Ready for Review
**Effort:** M (3 points)
**Dependencies:** 11.1 (wallets table must exist)
**PRD Feature:** F-022

## Description

As a developer,
I want `wallet_id` added as a foreign key to all portfolio data tables (`asset_types`, `asset_groups`, `assets`, `questionnaires`, `asset_scores`, `contributions`),
so that every piece of portfolio data is associated with a specific wallet for data isolation.

The migration follows a safe 3-step approach (ADR-011): add nullable FK → backfill → make NOT NULL.

[Source: PRD-v2 §3 Module 11, Feature F-022; ADR-011 (Safe Migration Strategy)]

## Acceptance Criteria

- [x] Given the migration is applied (step 1), when tables are inspected, then `wallet_id UUID REFERENCES wallets(id) ON DELETE CASCADE` column exists on: `asset_types`, `asset_groups`, `assets`, `questionnaires`, `asset_scores`, `contributions`
- [x] Given existing data in the tables, when the backfill migration runs (step 2), then a default wallet "Minha Carteira" is created for each distinct `user_id`, and all rows are assigned the corresponding `wallet_id`
- [x] Given the backfill is complete, when the NOT NULL migration runs (step 3), then `wallet_id` becomes NOT NULL on all 6 tables
- [x] Given the `price_cache`, `exchange_rates`, and `price_refresh_log` tables, when inspected, then they do NOT have a `wallet_id` column (prices are global)
- [x] Given the UNIQUE constraints, when inspected, then they are updated to include `wallet_id` where applicable (e.g., `asset_types` unique on `(user_id, wallet_id, name)`)
- [x] Given the migration, when applied to a database with existing data, then 100% of existing data is preserved with no data loss

## Dev Notes

- **Primary files:**
  - `supabase/migrations/013_add_wallet_id_nullable.sql` (step 1)
  - `supabase/migrations/014_backfill_wallet_id.sql` (step 2)
  - `supabase/migrations/015_wallet_id_not_null.sql` (step 3)
- **Constraints:** ON DELETE CASCADE for all wallet_id FKs (PRD-v2 §5)
- **Constraints:** Composite indexes on `(user_id, wallet_id)` for high-volume tables
- **Constraints:** Migration must be reversible at each step

- [Ref: ADR-010 (schema), ADR-011 (safe migration), ADR-014 (data isolation)]
