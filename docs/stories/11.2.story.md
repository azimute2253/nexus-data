# Story 11.2 — Add `wallet_id` FK to All Data Tables

**Epic:** 11 — Multi-Wallet Foundation
**Status:** Draft
**Effort:** M (3 points)
**Dependencies:** 11.1 (wallets table must exist)
**PRD Feature:** F-022

## Description

As a developer,
I want `wallet_id` added as a foreign key to all portfolio data tables (`asset_types`, `asset_groups`, `assets`, `questionnaires`, `asset_scores`, `contributions`),
so that every piece of portfolio data is associated with a specific wallet for data isolation.

The migration follows a safe 3-step approach (ADR-011): add nullable FK → backfill → make NOT NULL.

[Source: PRD-v2 §3 Module 11, Feature F-022; ADR-011 (Safe Migration Strategy)]

## Acceptance Criteria

- [ ] Given the migration is applied (step 1), when tables are inspected, then `wallet_id UUID REFERENCES wallets(id) ON DELETE CASCADE` column exists on: `asset_types`, `asset_groups`, `assets`, `questionnaires`, `asset_scores`, `contributions`
- [ ] Given existing data in the tables, when the backfill migration runs (step 2), then a default wallet "Minha Carteira" is created for each distinct `user_id`, and all rows are assigned the corresponding `wallet_id`
- [ ] Given the backfill is complete, when the NOT NULL migration runs (step 3), then `wallet_id` becomes NOT NULL on all 6 tables
- [ ] Given the `price_cache`, `exchange_rates`, and `price_refresh_log` tables, when inspected, then they do NOT have a `wallet_id` column (prices are global)
- [ ] Given the UNIQUE constraints, when inspected, then they are updated to include `wallet_id` where applicable (e.g., `asset_types` unique on `(user_id, wallet_id, name)`)
- [ ] Given the migration, when applied to a database with existing data, then 100% of existing data is preserved with no data loss

## Dev Notes

- **Constraints:** Composite indexes on `(user_id, wallet_id)` for high-volume tables
- **Constraints:** Migration must be reversible at each step
