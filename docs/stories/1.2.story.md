# Story 1.2 — Row Level Security (RLS) Policies

**Phase:** 1 — Foundation
**Status:** Draft
**Effort:** S
**Dependencies:** 1.1 (schema must exist before applying RLS)

## Description

As a developer,
I want RLS enabled on all 8 Nexus Data tables with `auth.uid()` policies,
so that no data is accessible without authentication and each user can only access their own data.

RLS is a PRD non-negotiable requirement (F-008 AC2). Every table must have policies for SELECT, INSERT, UPDATE, and DELETE restricted to `auth.uid() = user_id`.

[Source: PRD §5 Technical Constraints — Security, Feature F-008 AC2]

## Acceptance Criteria

- [ ] Given each of the 8 tables, when RLS is inspected, then `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` is active
- [ ] Given each table, when a policy is inspected, then it uses `USING (auth.uid() = user_id)` for SELECT/UPDATE/DELETE and `WITH CHECK (auth.uid() = user_id)` for INSERT/UPDATE
- [ ] Given an unauthenticated request (anon key, no session), when querying any table, then zero rows are returned
- [ ] Given an authenticated request from user A, when querying assets, then only rows with `user_id = A` are returned
- [ ] Given the `price_cache` table, when the Edge Function cron writes with service role key, then RLS is bypassed for UPSERT operations
- [ ] Error: Given a request attempting to INSERT with a different `user_id`, when RLS evaluates `WITH CHECK`, then the insert is rejected with a PostgreSQL RLS violation error (code 42501)
