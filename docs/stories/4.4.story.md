# Story 4.4 — Rebalance Orchestrator + Parity Test Suite

**Phase:** 4 — Rebalancing Algorithm
**Status:** Ready for Review
**Effort:** M
**Dependencies:** 4.1 (L1), 4.2 (L2), 4.3 (L3), ADR-004 (Rebalancing Algorithm Architecture)

## Description

As a developer,
I want the top-level `rebalance()` function that orchestrates L1→L2→L3 and a parity test suite comparing against spreadsheet reference data,
so that the complete algorithm is callable as a single function and validated against the existing spreadsheet.

The `rebalance()` function is the public API. It chains L1→L2→L3 and returns a structured `RebalanceResult` with per-asset allocations. The parity test suite uses at least 3 reference scenarios extracted from the actual spreadsheet.

[Source: PRD §3 Module 2, Feature F-002 AC2/AC7]

## Acceptance Criteria

- [x] Given the function signature `rebalance(portfolio, contribution)`, when called, then it returns a `RebalanceResult` with per-asset: ticker, shares_to_buy, amount_brl
- [x] Given R$12,000 and the same prices as the spreadsheet, when `rebalance()` runs, then output matches spreadsheet within < R$1 per asset tolerance
- [x] Given the result, when accessed programmatically, then `RebalanceResult` contains nested arrays: `types[].groups[].assets[]` with amounts at each level summing to the total contribution
- [x] Given at least 3 reference test scenarios from the spreadsheet, when run as Vitest tests, then all pass with < R$1 tolerance
- [x] Given the test suite, when executed on CI, then it runs automatically on every build
- [x] Given edge cases (zero contribution, all overweight, single asset in group, score of 0, negative group total score), when tested, then all return a `RebalanceResult` where `sum(allocations) <= contribution` and no exception is thrown
- [x] Error: Given invalid portfolio data (missing types, empty groups), when `rebalance()` is called, then it throws a descriptive validation error
