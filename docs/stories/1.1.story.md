# Story 1.1 — Supabase Database Schema Setup

**Phase:** 1 — Foundation
**Status:** Draft
**Effort:** M
**Dependencies:** None (first story — foundational)

## Description

As a developer,
I want the complete Supabase database schema (8 tables + 1 materialized view) created with proper types, constraints, and foreign keys,
so that all subsequent features have a reliable data layer to build upon.

The schema covers the entire portfolio domain: asset types (10 classes), asset groups (~15), individual assets (131+), price cache, questionnaires, scores, exchange rates, and future contributions. All tables include a `user_id` column for RLS enforcement.

[Source: PRD §5 Technical Constraints — Database Schema, Features F-001 through F-011 (all depend on schema)]

## Acceptance Criteria

- [ ] Given the Supabase project, when migration files are applied, then all 8 tables are created: `asset_types`, `asset_groups`, `assets`, `price_cache`, `questionnaires`, `asset_scores`, `contributions`, `exchange_rates`
- [ ] Given the schema, when foreign keys are inspected, then `asset_groups.type_id` references `asset_types.id` (ON DELETE CASCADE), `assets.group_id` references `asset_groups.id` (ON DELETE CASCADE), and `asset_scores.asset_id` references `assets.id` with `asset_scores.questionnaire_id` referencing `questionnaires.id`
- [ ] Given the schema, when inspected, then `price_cache.ticker` is PRIMARY KEY with `NUMERIC(15,4)` for price, and all tables have a `user_id UUID NOT NULL REFERENCES auth.users(id)` column
- [ ] Given the `asset_types` table, when inspected, then it has columns: `id`, `name`, `target_pct NUMERIC(5,2)`, `sort_order INTEGER`, `user_id`, `created_at`, `updated_at`; and given the `assets` table, it has: `id`, `ticker`, `sector`, `quantity NUMERIC(15,6)`, `group_id`, `price_source TEXT`, `is_active BOOLEAN DEFAULT true`, `manual_override BOOLEAN DEFAULT false`, `whole_shares BOOLEAN DEFAULT true`, `user_id`, `created_at`, `updated_at`
- [ ] Given the `portfolio_summary` materialized view, when queried, then it returns aggregated total value per asset type by joining `assets`, `price_cache`, and `exchange_rates`
- [ ] Given the migration files, when committed to git, then they are located in `supabase/migrations/` directory and are idempotent (re-runnable without errors)
- [ ] Error: Given a duplicate ticker in `price_cache`, when an INSERT is attempted, then UPSERT (`ON CONFLICT`) updates the existing row without raising an error
- [ ] Error: Given a migration applied to a schema with existing tables, when re-run, then it completes with no duplicate-object errors (idempotent via `IF NOT EXISTS`)
