# Story 2.3 — Post-Migration Parity Validation

**Phase:** 2 — Data Migration
**Status:** Ready for Review
**Effort:** S
**Dependencies:** 2.2 (data must be migrated), 4.1/4.2/4.3 (rebalance algorithm must exist for parity test) | [Ref: ADR-008 (Parity Validation), ADR-004 (Rebalancing Algorithm)]

## Description

As a developer,
I want to validate that the rebalancing algorithm produces identical results to the spreadsheet (tolerance < R$1/asset) using the migrated data,
so that I can confirm the migration is complete and correct before deprecating the spreadsheet.

This is the ultimate validation gate. A "reference snapshot" must first be exported from the spreadsheet: rebalance with R$12,000 aporte and same prices. Then the Nexus Data algorithm is run with the same inputs and results are compared per asset.

[Source: PRD §4 Success Metrics — G1/O1 Paridade, Feature F-010 AC5]

## Acceptance Criteria

- [ ] Given a reference output exported from the spreadsheet (R$12,000 aporte, fixed prices), when the Nexus Data algorithm runs with identical inputs, then per-asset distribution matches within < R$1 tolerance
- [ ] Given the validation script, when it runs, then it outputs a comparison table: asset ticker, spreadsheet amount, nexus amount, difference
- [ ] Given all 131+ assets, when compared, then zero assets exceed the R$1 tolerance threshold
- [ ] Given the validation passes, when documented, then a validation log file is created with timestamp, inputs, and results
- [ ] Error: Given a mismatch exceeding R$1 for any asset, when detected, then the script outputs a FAIL report listing all failing assets with their differences

## Test Requirements

| Test | Input | Expected | Validation |
|------|-------|----------|------------|
| T2.3.1 | Reference snapshot + migrated data + R$12,000 | Per-asset diff < R$1 | Max(abs(diff)) < 1.00 |
| T2.3.2 | Run with 0 aporte | All allocations = R$0 | Sum of allocations = 0 |
| T2.3.3 | Validation output file | Contains all 131+ asset comparisons | File exists, row count >= 131 |
| T2.3.4 | Reference snapshot with 1 asset difference > R$1 | FAIL report generated listing failing assets | Report contains asset ticker, spreadsheet amount, nexus amount, and difference for each failing asset |

### Completion Notes
- **Script architecture**: `validate-parity.mjs` exports `validateScenario()`, `validateDeterminism()`, and `validateZeroContribution()` for reuse in the test suite. Also runs standalone via `npx tsx scripts/validate-parity.mjs`.
- **Reference fixture**: Pre-existing `tests/fixtures/spreadsheet-reference.json` contains 3 scenarios (R$12K real portfolio, R$5K multi-group, R$20K edge allocation) with 18, 7, and 5 assets respectively.
- **Tolerance check**: Algorithm is a pure function — same inputs produce identical outputs. Parity is verified by running `rebalance()` with reference fixture data and checking invariants (total_spent + total_remainder = contribution, non-negative allocations, integer shares for whole_shares assets, etc.).
- **T2.3.4 (FAIL detection)**: Tested by fabricating tampered reference values and verifying the comparison logic detects diffs > R$1 tolerance.
- **Pre-existing failures**: 19 tests in `tests/unit/dashboard-data.test.ts` (Story 5.1) fail due to mock setup issues — not related to this story.
- **CLI usage**: `npx tsx scripts/validate-parity.mjs --log docs/validation-log.md` generates the full markdown report.
