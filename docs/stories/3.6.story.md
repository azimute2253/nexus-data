# Story 3.6 — Auto-refresh on Portfolio Entry

**Epic:** 3 (Price Engine)  
**Size:** S (~30 min)  
**Dependencies:** 3.4 ✅, 9.2 ✅  
**Wave:** 6 Track F  
**Status:** Ready for Review

## Context

Users want fresh prices when viewing the portfolio dashboard, but running a cron every 5 minutes burns API budget unnecessarily when the dashboard is idle. This story implements **on-demand refresh triggered by portfolio page entry** with a configurable cooldown to balance freshness with rate limit conservation.

**Related ADR:** [ADR-005 Addendum](../adr/ADR-005-price-caching-strategy.md#addendum-story-36--auto-refresh-on-portfolio-entry-2026-03-22)

---

## User Story

**As a** portfolio user  
**I want** prices to refresh automatically when I open the portfolio page  
**So that** I see current data without manual "Refresh" clicks, but API budget isn't wasted on empty refreshes

---

## What to Implement

### 1. Database Schema

Create `price_refresh_log` table to track last refresh timestamp:

```sql
CREATE TABLE price_refresh_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  refreshed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  user_id UUID REFERENCES auth.users(id) -- optional: global or per-user
);

CREATE INDEX idx_price_refresh_log_refreshed_at ON price_refresh_log(refreshed_at DESC);
```

### 2. Portfolio Page Logic

In `src/pages/dashboard/portfolio/index.astro` (or server component):

```typescript
// 1. Check last refresh
const COOLDOWN_HOURS = process.env.PRICE_REFRESH_COOLDOWN_HOURS || 4;

const { data: lastRefresh } = await supabase
  .from('price_refresh_log')
  .select('refreshed_at')
  .order('refreshed_at', { ascending: false })
  .limit(1)
  .single();

const now = new Date();
const lastRefreshTime = lastRefresh ? new Date(lastRefresh.refreshed_at) : null;
const hoursSinceRefresh = lastRefreshTime 
  ? (now.getTime() - lastRefreshTime.getTime()) / (1000 * 60 * 60)
  : Infinity;

// 2. Trigger refresh if cooldown expired
if (hoursSinceRefresh >= COOLDOWN_HOURS) {
  // Show loading state
  const refreshing = true;
  
  // Call refresh API
  await fetch('/api/prices/refresh', { method: 'POST' });
  
  // Log new refresh
  await supabase
    .from('price_refresh_log')
    .insert({ refreshed_at: now.toISOString() });
}

// 3. Load prices (now fresh or from cache)
const { data: prices } = await supabase.from('price_cache').select('*');
```

### 3. API Route (if not exists)

`/api/prices/refresh` should:
1. Call price engine (Story 3.4)
2. Update `price_cache` table
3. Return summary: `{ updated: 131, failed: 0, duration_ms: 8432 }`

### 4. Configuration

Add to `.env`:
```
PRICE_REFRESH_COOLDOWN_HOURS=4
```

### 5. Loading State

Show spinner or skeleton while refresh is in progress:
```tsx
{refreshing && (
  <div className="refresh-banner">
    🔄 Atualizando preços... (~10s)
  </div>
)}
```

## Acceptance Criteria

### Functional

- **AC1:** Refresh triggers on portfolio entry if `(now - last_refresh) > COOLDOWN_HOURS`
- **AC2:** Cooldown period configurable via `PRICE_REFRESH_COOLDOWN_HOURS` env var (default 4h)
- **AC3:** Refresh timestamp logged to `price_refresh_log` table
- **AC4:** No refresh if within cooldown window (reads cached prices immediately)
- **AC5:** Loading state shown during refresh (~10s)
- **AC6:** Works for multiple users (global cooldown or per-user scope)

### Technical

- **AC7:** `price_refresh_log` table created with migration
- **AC8:** `/api/prices/refresh` endpoint exists and returns success/failure summary
- **AC9:** Type check passes (`tsc --noEmit`)
- **AC10:** Build passes (`npm run build`)

### Edge Cases

- **AC11:** First-ever entry (no `price_refresh_log` rows) → triggers refresh
- **AC12:** Concurrent page loads (2 users enter < 1 min apart) → only 1 refresh (lock mechanism or accept duplicate)

## Trade-offs

| Pro | Con |
|-----|-----|
| Prices refresh only when needed | First user in 4+ hours waits ~10s |
| Conserves API budget | Doesn't account for market hours (may refresh at 2am) |
| Simpler than cron (no market hours logic) | Requires client loading state |
| Configurable cooldown adapts to usage | Multiple concurrent loads may duplicate refresh |

---

## Out of Scope

- Market hours awareness (use simple time-based cooldown)
- Per-user refresh cooldown (use global timestamp)
- Lock mechanism for concurrent refreshes (accept duplicate if < 1 min apart)
- Cron-based scheduled refresh (Story 3.4 or future enhancement)
