# Story 1.3 — Auth Integration with azimute-blog

**Phase:** 1 — Foundation
**Status:** In Review
**Effort:** S
**Dependencies:** 1.2 (RLS must be active before auth routes serve data)

## Description

As a user,
I want to log in with my existing azimute-blog email/password credentials and have all `/dashboard/portfolio/*` routes protected,
so that my financial data is secure and I don't need a separate login.

The Nexus Data pages are embedded in azimute-blog (ADR-001). Auth middleware already protects `/dashboard/*` routes. This story ensures the Supabase client is configured with `@supabase/ssr` for the Nexus Data pages and that session persistence via refresh tokens is functional.

[Source: PRD §3 Module 6, Feature F-008]

## Acceptance Criteria

- [ ] Given an unauthenticated user, when navigating to `/dashboard/portfolio`, then they are redirected to the login page
- [ ] Given a user with valid email/password, when logging in, then they receive HTTP 302 redirect to `/dashboard/portfolio` and the page renders with the authenticated user's email visible in the UI
- [ ] Given a logged-in user, when they close the browser and reopen the next day, then the session is still valid (persistent via Supabase refresh tokens) — F-008 AC3
- [ ] Given the Supabase client in Nexus Data pages, when initialized, then it uses `@supabase/ssr` for server-side session validation
- [ ] Given the client-side JavaScript bundle, when inspected, then only `SUPABASE_ANON_KEY` is present and `SUPABASE_SERVICE_ROLE_KEY` does not appear in any client-accessible code — F-008 AC4
- [ ] Error: Given an expired session with no valid refresh token, when accessing any dashboard route, then the user receives HTTP 302 redirect to the login page
